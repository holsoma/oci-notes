Security

  • shared security model is heard often, what does this mean?
    • on prem, you own the whole stack and are responsible for end-to-end security
    • as you move to the cloud, some of the responsibilities transfer to the cloud provider
    • cloud will be responsible for virtualization, physical hosts, physical network, physical datacenter
  • oracle security is in depth, and available at different layers of the stack
    • infrastructure protection
      • use cases - network security controls, filter malicious web traffic, DDoS protection
      • using - DDoS protection services, web app firewall (WAF), security lists
      • WAF helps with layer 7 DDoS protection, layer 3 and 4 DDoS protection is enabled by default
    • OS and workload protection
      • use cases - patch management, workload isolation, managed bastion
      • using - bare metal servers with dedicated hosts (single tenant model), bastion services and OS management service (automate patches and simplifies package deployment)
    • detection and remediation (cloud security posture management services CSPM)
      • use cases - security posture management, security advisor, secure enclave, automated security assessment
      • using - cloud guard and security zones
    • data protection
      • use cases - encryption for data at rest and in transit, key storage and management, rotate manage and retrieve secrets
      • using - vault (key management system), data safe, default encryption
    • identity and access management
      • use cases - manage user access and policies, MFA, single sign-on to identity providers
      • using - IAM authorization and things like MFA
  • security is not just one service or add on, it's a whole extensive set of services available in different layers of the stack

Cloud Guard

  • helps to monitor and identify potential security issues and remediate them (it can completely automate it!)
  • detect problems (check configuration, monitor activities) -> apply response (correlate problems, apply fix)
  • how does this work?
    • targets - resources to be examined (eg. target: compartments)
    • detectors - cloud guard components that identify issues by flagging them (eg. there is a public bucket)
    • problems - potential security issues notifications that a configuration or activity is an issue
    • responders - act on potential security issues (eg. disable public bucket, stop public instance)

Security Zones and Security Advisor

  • security zone is to configure a location that you cannot disable security (cannot be without security)
  • security advisor is a service that unifies security zone, cloud guard and other capabilities
  • what are security zones?
    • you have resources in your compartments A and B, you could designate compartment B as a security zone
    • in this security zone, nothing but your policies get enforced here (when there is a policy violation, that operation is denied)
    • networking, storage, compute and databases are supported
    • if you define that a subnet always has to be private, a public subnet creation will be denied
  • what is security advisor?
    • cloud guard + security zone + other security services
    • supports object storage, file storage, block volume and VMs
    • eg. security advisor will walk you through how to create a bucket in security zone + give you some requirements

Vulnerability Scanning

  • routinely checks hosts for potential vulnerabilities
  • common vulnerabilities and exposures (CVE) - known database of vulnerabilities, each assigned a unique id
  • how it works - you create recipes and define parameters for the recipe
    • define daily or weekly
    • port scanning - check for 1000 most common ports and close unintentional open ports
    • agent-based scanning - checks against standard benchmarks published by center for internet security (CIS), then assigns a risk level
      • if it meets 20% assign a score, if it meets 40% assign a score, and so on
  • reports and risk levels are generated, also tells you things like which OS packages require updates and gives you details on OS configs that hackers might exploit
  • recap
    • vulnerability scanning service is designed to check for vulnerabilities and deviations from best practices for compute instances and OS

Vault

  • service to centrally manage encryption keys and secret credentials (removes the need to store these keys in code)
  • keys - specifies how to transform plaintext into cipher text and back during encryption
  • secrets - credentials like passwords, certificates, SSH keys, auth tokens
  • two type of protection modes for keys - software and hardware security modules (HSM)
    • HSM
      • master encryption key is stored on HSM device and cannot be exported
      • all cryptographic operations happen on the HSM as well
    • software
      • master encryption key is stored on a server and can be exported
      • cryptographic operations can be performed on the client instead of on the server (compute or storage host)
  • algorithms vault supports - AES, RSA and ECDSA algorithms
    • AES
      • symmetric algorithm - same key encrypts and decrypts data
    • RSA
      • asymmetric algorithm - public key encrypts data and private key decrypts data
    • ECDSA
      • keys used in digital signing, cannot be used to encrypt or decrypt data
  • keys are integrated in OCI, you can rotate your master keys (you don't ever have to do a complete set of encryption again)
  • it is a regional service, can be accessed with a public API endpoint
  • vault operates using envelop encryption
    • a two-tier hierarchy for keys (master key to encrypt/decrypt data key, data key to encrypt decrypt your data [block storage])
    • you can use IM policies to authorize access to master keys
    • you can do audit logs to monitor key related activities as well
    • benefits?
      • easier to manage
      • limits blast radius
      • you can just rotate master keys (don't have to do complete data re-encryption)
    • however...
      • if master key is deleted, you can't recover data (soft delete keys with a seven day gap)
      • keep in mind - vault cannot be deleted immediately (you can schedule the deletion by configuring a waiting period)
      • vault cannot be recovered after deletion, that's why there's a 7-30 day delete process

Web Application Firewall

  • firewalls operate at layers 3,4 and 5 - they can understand IP, TCP, UDP and sessions (but cannot understand layer 7)
  • WAP protects against a complex layer 7 or application layer attacks
    • attacks like SQL injection, cross site scripting
  • what is WAP?
    • a device, server side plugin or filter that applies to HTTP or HTTPS traffic
    • intercepts this traffic and passes them through a set of filter and rules, a WAP is able to uncover and protect against attack streams
    • a typical response will
      • allow a request to pass through, audit and log the request
      • block the request
  • WAP capabilities
    • supports over 250 rule sets to protect against various web attacks (SQLi, XSS, STML injections)
    • javascript challenge, captcha challenge, device fingerprint challenge, whitelisting capabilities - mitigate bad bots and allow legitimate traffic only
    • user access controls can be customized (filter based on location)
    • multi-cloud support - can be on OCI, on prem, or across cloud deployments

Bastion

  • a fully managed service you provision inside your VCN
  • provides seamless RDP/SSH connectivity to VMs
  • provides time-limited access to target resources
  • how does it work?
    • bastions are logical entities that provide secure public access to target resources in the cloud, that you otherwise cannot reach from the internet
    • these resources can be your DB, compute instance (and are located in a private subnet)
    • instead of using a NAT gateway to go to the internet and get patches (you would have to manage the NAT gateway), you can reach those instances using the bastion
    • bastions, because they do these kind to reaching to instances in a private subnet, have to be in a public subnet
    • how can you then, secure the bastion?
      • you can integrate with the IM service - decide who can access the bastion in the first place and what kind of permissions they have
      • configure a CIDR block allow list - this list specifies what IPs can connect to a bastion (if IP not in allow list, SSH command fails)
  • recap
    • bastion provides restricted access to target resources
    • removes the need to create and maintain your own bastions (which can lead to potential security breaches)
    • hence, service is fully managed by oracle (it is also free!)
