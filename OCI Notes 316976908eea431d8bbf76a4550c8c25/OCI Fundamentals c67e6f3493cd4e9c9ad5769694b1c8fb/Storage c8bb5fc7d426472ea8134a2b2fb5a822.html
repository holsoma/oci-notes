Storage

When deciding what type of storage to use, ask these questions:

  • persistent or non-persistent storage
  • what type of data files are you storing?
    • database files
    • videos
    • audio
    • photos
    • text
  • what kind of performance do you need?
  • what kind of capacity do you need?
  • how much input/output per second (IOPS)?
  • how much is the throughput?
  • decide on durability
    • how many copies of data do you want to maintain
  • persistence vs durability
    • persistence means you safely store the data
  • decide on connectivity
    • local vs network
    • how does the app access the data
  • protocol
    • block vs file vs HTTP

Block Volume

  • provide persistent and durable storage, you can
    • create, attach disks
    • detach, delete disks
    • keep data even after you delete instance
  • block volume tiers
    • basic - large sequential workloads, 2 IOPS/GB
    • balanced - balanced choice for random IO, 60 IOPS/GB
    • higher performance - most IO demanding workloads, 75 IOPS/GB
    • ultra high performance - highest IO demanding workloads, 90-225 IOPS/GB
  • encrypted block volumes, BYOK, or in-transit encryption
  • feature: read/write shareable, VMs can read and write data from a single block volume
  • feature: resizing block volumes on demand
  • feature: replicating block volumes across regions for
    • disaster recovery
    • migration
    • business expansion
  • volume groups, group volumes together for easy management

File Storage

  • hierarchical collection of documents organized into named directories
  • supported by all major OS and hypervisors, including UNIX and Windows
  • network file system (NFS) for UNIX and server message block (SMB) for Windows
  • allows creation, deletion, reading, writing, sharing and locking
  • (typically) no extra client software needed
  • provides access across networks
  • shared storage for compute instances
  • supports NFS v.3 distributed FS
  • data protection in the form of snapshots
  • security: data-at-rest and in-transit encryption

FSS Use Cases

  • oracle applications
  • general purpose FS
  • microservices containers
  • scale out apps
  • analytics

Object Storage

  • internet-scale, high-performance
  • data managed as objects
    • object + object metadata in a bucket in a namespace
    • bucket
      • unique name in a tenancy
      • flat hierarchy
    • namespace
      • logical entity, top level container for all buckets/objects
      • global unique name
    • /o - object, /b - bucket, /n - namespace, us-sanjose-1 - region
  • ideal for unstructured data
  • regional, public service
  • multiple storage tiers
    • standard storage tier
      • fast, immediate and frequent access
      • most recent copy of data
      • instant retrieval
      • can't be downgraded
    • infrequent access storage tier
      • ideal for data you access infrequently
      • storage costs lower than standard storage tier
      • minimum retention requirement: 31 days
      • retrieval fees
    • archive storage tier
      • seldom or rarely accessed data
      • minimum retention requirement: 90 days
      • objects need to be restored before download
      • restore time: 1 hour
      • download time: 24 hour
      • archive bucket can't be upgraded
    • feature: auto tiering, move between tiers
    • feature: object lifecycle management, transition data to lower-cost tier or delete them using rules
    • feature: objects automatically versioned
    • feature: data encryption, you can BYOK
    • easy to access stored data with a single API call
  • private access form OCI resources (eg. compute)
  • advanced capabilities

OSS Scenarios

  • as a content repository
  • to store unstructured and semi-structured data
  • big data/spark/hadoop/data analytics
  • archive/backup

OCI Data Migration services

  • data transfer disk (offline)
    • ship your own disk up to 100 TB
    • data transfer site
    • data encrypted
    • data uploaded > verified > erased > returned to customer
  • data transfer appliance (offline)
    • appliance is sent to you, might be more applicable because you don't have to send multiple drives
    • 150TB of usable space
    • data transfer site
    • data encrypted
    • data uploaded > verified > erased from DTA (NIST 800-88 media sanitization guideline)
  • storage gateway service (online)
    • linux-docker appliance installed on one or more hosts in your on-premises data center
    • exposes a NFS mount point that can be mounted to any supported NFS client host
    • this mount point maps to an object storage bucket on the OCI site
    • files in SG is written to as an object with the same name
    • you can use object storage or archive storage
      • great for data migration
      • can be used for cloud tiering
      • disaster recovery or backup
