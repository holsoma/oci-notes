Governance and Administration

Pricing

  • transparent pricing
  • lower pricing than competitors
  • what kind of pricing do we support?
    • pay as you go pricing - only charged for resources you consume, no minimum service period, usage is metered for
    • consumption based pricing - charged only when you consume the resource (for Functions)
      • different from pay as you go because you have to pay for the VMs even though you might not be using them or they might be running idle
    • annual universal credits - you commit to an annual pool of funds, and u get significant savings because of this (but u have to use these credits within 12 months)
      • pay as you go, if you end up with more resource usage (discounts vary based on size and terms of deal)
    • bring your own license (BYOL)
      • this is for customers that are running on prem licenses, and want to reuse that in the cloud
        • why? - because it reduces overall cost, and you can reuse those licenses in cloud
  • factors that impact pricing
    • size of resource - bigger resources cost more (this is natural, because service pricing is based on consumption)
    • data transfer - no ingress cost, careful with egress cost
      • can cost up to 1/4 of your bill or even higher, because your applications are always communicating, whether to end users or internally
      • incoming traffic is free, outgoing traffic is 10 times lower than other cloud providers
    • resource type - VMs vs bare metal have different price points, serverless and BYOL has different price points as well
    • note: pricing is the same across regions, this is different than traditional cloud models where pricing is very local

Cost Management

  • OCI budgets
    • use budgets to track costs in tenancy
    • create budget for compartment, set up alerts that will notify you if a budget is forecast to be exceeded
  • cost analysis
    • analyze your cost after you have spent it
    • nice way to look at past costs and make changes for the future
  • usage reports
    • downloaded in the form of CSV files, generated daily
    • show usage data for each resource in your tenancy
    • CSV files stored in an object storage bucket, accessible using a cross-tenancy policy
      • allows you to look across multiple tenancies if there are multiple tenancies being used
  • service limits
    • set limits, quota and usage
    • set by oracle based on the pricing model
    • limit how many resources you can run in a traditional tenancy to prevent things like fraud and misuse (you can look at your current limits and change it with a support request)
  • compartment quotas
    • you can write different policies, setting the maximum number of cloud resources that can be used for a compartment (eg. use 10 VMs only)
      • prevent misuse and save cost

Tagging

  • key-value pairs (key is for the tag, value is the value you want to assign), to better organize your resources
  • check service limits for how many tags to put on cloud object
  • why use tags?
    • to organize your cloud resources (hundred and thousands of them)
    • cost management - you can pull up resources by the tag, and you can find out what is the cost usage for these resources (find out how much application is consuming)
  • tag-based access control
    • control access based on tags for the resources, not the users and groups
    • this is powerful, because you can write policies based on tags

Two kinds of tags

  • free form tags
    • key and a value - key is environment, value is production
    • no defined schema, no kind of access restriction
  • defined tags
    • contained within namespace (namespace = operations)
    • you can put as many tags as you want in those namespaces (a container for a set of tags with tag key definitions)
    • defined schema, secured with policy (you can only allow a particular group of operations admin to apply that particular tag)

Tag Namespace

  • you can specify the type of values allowed (type = string), you could set a specific set of values for users to choose from when they come to apply this particular tag
  • keep in mind that you cannot delete tag namespace after you define it (you can retire it though) - more for governance purposes, you can keep an audit of the namespaces you had in your account
  • another way is you could first determine who applies the tag by writing the policies, then specify what kind of values are allowed
