App Development

  • cloud-native vs cloud-enabled vs cloud-based application
  • cloud native
    • constructed on the cloud from scratch
    • harness the power of public cloud platforms (elasticity, scalability, resiliency, flexibility)
  • cloud enabled
    • application that has been moved to the cloud, but originally developed for use in an on-premises environment (typically legacy applications)
  • cloud based
    • middle ground between cloud native and cloud enabled
  • cloud-native app building blocks
    • containers
      • packing your application code into one single executable unit
      • portable
    • microservices
      • split app feature or tasks into independent, loosely coupled services
      • allows you to design each service with a limited functional scope that focuses on one feature or capability
      • this improves app maintenance, feature development, testing, deployment, scalability
    • service meshes
    • immutable infrastructure
    • APIs
    • all these services create loosely coupled systems that are resilient, manageable, and observable, they allow developers to make high-impact changes frequently and with minimal effort
    • there is a 12-factor app methodology that details how modern, cloud-native apps should be built
  • OCI developer services (this will help when you build cloud-native apps)
    • container and artifacts
      • managing containerized applications, associated resources and deployments
      • OKE kubernetes service
      • container registry
      • artifact registry
      • service mesh service
    • functions
      • managed severless offering
    • API management
      • lets you define REST interfaces for other services
      • microservices you create can call each other and leverage this managed service
    • DevOps
      • visually script CI/CD
    • resource manager
      • service to deploy infrastructure as code with managed Terraform templates
    • developer resources
      • tools for writing apps that leverage OCI directly (terraform, ansible, powershell support, cli, etc.)

Oracle Container Engine for Kubernetes (OKE)

  • VMs vs Containers
    • VMs have a hypervisor (virtual machine monitor) like an ESXI and VMs running on top of the hypervisor, each VM has its own OS inside it, then the libraries and the dependencies and the application
    • Containers have the underlying hardware (the same as VMs), then the OS, then container runtime (like Docker) installed on the OS
    • the container runtime manages the containers that run with the application, libraries and dependencies, but there is no OS inside the containers
  • why containers over VMs?
    • VMs have an extra overhead, since there are multiple VMs running, multiple OS systems running, each with their own kernels, etc. -> higher utilization
    • VMs also use more disk space since
    • VMs take longer to boot up because each OS has to boot up
    • Containers boot up in a matter of seconds, they are lightweight (usually MBs in size, but that's because the OS is heavy), this means smaller images as well
    • Containers are portable, build file that goes with your image, you can deploy it anywhere (hence more suitable for cloud-native application's microservices!)
  • Kubernetes
    • containers need to talk to each other (to scale up or down, etc.)
    • this connectivity is known as container orchestration, and Kubernetes is an open source system for automating, deployment, scaling and management of containerized applications
  • advantages of Kubernetes
    • run containerized applications of any scale with no downtime
    • self-heal containerized application -> provides resiliency
    • auto-scale containerized applications -> ensure optimal utilization
    • all these simplifies deployment operations greatly
  • what is Oracle Container Engine for Kubernetes (OKE)?
    • fully managed, scalable, and highly available kubernetes system with many features for developers (like one click cluster creation), CLI/API support
    • support for running these on ARM based and GPU based instances
    • DevOps advantages like auto-scaling support
    • automatic kubernetes cluster upgrades
    • self-healing cluster nodes
  • how does OKE work?
    • a node pool which consists of nodes (a machine which kubernetes is installed) which consists of pods (a group of one or more containers with shared storage and network resources, a specification file for how to run containers inside the pod AKA smallest unit of compute in a managed kubernetes environment)
    • one highly available control plane nodes manage the worker nodes/node pools and pods in the cluster
    • you don't have to pay for control plane nodes
      • oracle manages control plane nodes, which have different components (kube-controller manager, cloud-controller manager, kube-API server, kube-scheduler, and etcd[key-value store used for kubernetes to back cluster data])
      • these components make decisions about the cluster (how to schedule containers, detection and response to cluster events: scale up, scale down, node failure)

Oracle Cloud Infrastructure Registry (OCIR)

  • highly available Docker v2 container registry service
  • stores docker images in private or public repositories
  • fully managed service on OCI
  • what problems does it solve?
    • hard for development and operations team to maintain consistent images for containerized applications
    • unable to enforce access rights and security policies for images
  • difficult to find the right images and have them available in region of deployment
  • what it basically does
    • makes it easy for you, as a developer to store, share and manage container images, such as Docker images
    • now you can use container registry as a:
      • private docker registry - push and pull docker images using standard docker cli
      • public docker registry - enable any user with internet access to pull images from public repositories in container registry
    • you can then deploy these images globally, in any region

Repositories

  • name in a container registry to push and pull images from
  • meaningfully named collection of related images grouped together (such as different versions of the same source image into same repository)
  • you access these images using an image path: <region-key>.ocir.io/<tenancy-namespace>/<repo-name>:<tag>
  • images to pull are specified in the application's manifest file, with a credential to use when pulling
  • fully integrated with OKE, and Functions API

DevOps

  • continuous integration and continuous delivery, CI/CD, best practice in DevOps
  • continuous integration
    • frequent, small code changes -> less likely to conflict with other code merges
    • error fixes continually integrated
    • automatic build and test steps
  • continuous delivery and deployment
    • once main code branch passes unit integration acceptance or other tests, it is released to production automatically (continuous deployment) or manually (continuous delivery)
    • continuous delivery is made possible with a deployment pipeline
      • every time change is committed to code repo -> tested -> marked as release-able or deployable

OCI DevOps Service

  • CI/CD platform for developers
  • automates delivery and deployments of software or workloads to OCI compute platforms
  • you can release features more frequently with fewer errors and less downtime
  • how does it work?
    • build and deployment pipeline are key components
    • build pipeline
      • build stage triggered when you commit -> fetches required libraries and assets from code -> creates build artifact (a software package, library, zip file) -> store in a container or artifact registry
      • rule of thumb: non-container, non-compute images or artifacts go into artifact registry (eg. zip files, libraries) VS docker image goes into container registry
    • deployment pipeline
      • artifacts are retrieved and deployed to staging environment -> integration tools are run -> approval -> deployment to production environment (eg. compute hosts - VM or bare metal machine, OKE container engine, or serverless offering like Functions)

APIs

  • REST (representational state transfer) or RESTful APIs
  • REST APIs are accessed over HTTP, are stateless, and are client-server communications
  • client and server can be written in different languages
  • APIs help clients access a microservice, but this can cause complexity when every client talks to every microservice
  • we can solve this by putting an API gateway in between clients and microservices -> clients connect to a single endpoint (API gateway), which is configured to forward requests to individual microservices

API Gateway

  • policy for who accesses what API is defined in the gateway, enforcing things like AuthN, SLAs (service level agreements) , and also does things like logging, metrics, monitoring, etc.
  • API gateway is where API is made available, so there is an endpoint on the gateway to determine whether request should go to back-end service to serve the request
  • API DOES NOT run on the gateway, but rather the endpoint to access the API is available on the gateway

API Gateway Service

  • in OCI, API gateway is a network attached device, much like a load balancer (it is serverless, you create it and it gets attached to your network)
  • fully managed by Oracle, security policy is enforced, we do monitoring, logging, metrics, some sort of mediation performed
  • features of OCI API gateway
    • regional service, appears at network device on customer's VCN
    • authZ
    • rate-limiting
      • limits the number of API calls to back-end servers
      • no scenarios where you overwhelm the server and DDOS attack happens
    • create private and public gateways, in private or public subnets
    • custom domain
    • monitoring, logging
    • integration with other services (including Oracle Integration Cloud)
    • supports Open API standards
    • built in mock testing
    • integrated with logging analytics
    • use case: API gateway becomes a front-end (facade pattern), or for mock testing and use with third party services
  • Remember
    • when defining an API, you need to define access, so you need some kind of policy, which is defined at the API gateway
    • API does not run on the gateway, the endpoint to access the API is available on the gateway

Functions

  • serverless offering
  • increasing abstractions, decreasing concern (and control) over infrastructure implementation
    • bare metal > VMs > container > functions
  • for Functions, you don't write code in a specific runtime, you don't worry above servers, about infrastructure, the cloud provider is responsible for executing your code
  • one of the greatest advantages: truly leads to a consumption based pricing model, where you are only billed for the time your function is running -> you go from a pay-as-you-go pricing to a completely consumption-based pricing (that's why more and more people use Functions)
  • Functions-as-a-service - you write code and code is executed, AKA event driven architecture
    • in reality, functions run inside container, you are billed only when the function runs
    • very capable of parallel execution
    • oracle functions is powered by Fn project open source engine (unlike other cloud vendors)
  • how does it work?
    • uploaded code and configs is packaged as a container image and stored in OCI registry -> you set up trigger actions (direct invoke: CLI or API; or OCI events) -> code runs when triggered -> pay for code execution time only
    • function integrations - monitoring, identity, registry, logging, network, OCI services

Resource Manager (ORM)

  • helps install, configure, and manage resources through the infrastructure as code model
  • think of it as a managed Terraform (infrastructure as code) service
    • you take your infrastructure components and describe them using a high level config syntax
    • this creates a blueprint of your infrastructure that can be versioned and treated like code
    • it also can be shared and reused
    • you can set up and destroy stacks - that's a big deal because it's hard to go back and figure out all the dependencies, set something up and destroy that setup
  • what are some benefits?
    • improves team collaboration
    • enables automation
      • config can be treated as code and automated
    • seamless integration with OCI platform
      • for eg. if you create a Kubernetes cluster on the OCI console, you can save as Terraform stack, and go on and apply that stack later on
    • provides a console UI experience
  • how does it work?
    • you upload a bundled terraform config file, your actual terraform code -> you create a stack -> run terraform action against that stack
    • your source can be anywhere: .zip, template, compartment, bucket, folder, git
    • create a stack (a distinct set of cloud resources) from source -> jobs (standard terraform commands: plan, apply and destroy) are executed against a stack -> refreshed terraform state is securely managed and then refreshed for each terraform action coordinated by resource manager (this ensures a single source of truth for current infrastructure state) [one of the big advantages over doing it yourself]
  • Recap
    • ORM is a managed terraform service
    • helps with automating deployments and sharing deployments with the wider team
