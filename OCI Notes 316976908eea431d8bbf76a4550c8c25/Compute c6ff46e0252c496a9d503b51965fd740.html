Compute

Compute

  • virtual machines and bare metal servers
  • 3 defining characteristics: scalability, high performance, lower pricing
  • flexible shapes
    • choose own cores, processors, memory
    • flexibility in choosing your own configuration
    • flexible sizing: small, medium, large (no need to change machine types)
  • bare metal
    • full machine, server, completed dedicated
  • dedicated host
    • full dedicated bare metal machine, but you can run VMs as well
  • VMs vs dedicated host
    • VMs are shared and multi-tenant
    • dedicated host to run own VMs
  • leverage choice of processors - AMD, Intel, and Ampere (Arm-based processor)
  • industry-leading price performance with Arm - higher price performance with Ampere Altra over AMD Milan
  • 50% cheaper than other cloud, pay for what you use
  • you can use preemptible VMs for further savings, these VMs are
    • low cost
    • short lived VMs
    • used for batch jobs
    • have fault tolerant workloads
    • 50% cheaper than other instances

Instance Basics

  • an instance is a compute host
  • compute host has several dependencies
  • to have an instance you need a VCN, subnets in the VCN
  • compute host is a physical construct, networking is a virtual construct
  • they are related - compute host has a network interface card are we virtualize that card (VNIC), and the VNIC is in the subnet (that's the association)
  • private IP for compute host comes from VNIC
  • another set of dependencies compute instances has
    • boot volume
    • boot disk
    • block volume
  • image (from boot disk) used to boot up instance defines the OS
  • data disks live on the network as well, works together with OS disks using the block volume service which the compute hosts uses to run the OS

Recap

  • compute control plane has dependency on the networking control plane and the block storage control plane
  • live migration between hosts is important as well to ensure that your applications are also running even during maintenance events

Scaling

  • vertical scaling
    • scaling instance shape up or down
    • new shape must have same hardware architecture
    • downtime required
  • good practice: stop instance before scaling
  • horizontal scaling/autoscaling
    • add or take away VMs of the same shape (scale-out or scale-in)
    • enables large scale deployment of VMs
    • high availability
    • match traffic demand by adding/removing VMs automatically
    • no extra cost for using autoscaling
  • three steps to get autoscaling
    • have a running instance, create a config (OS image, metadata, shapes, VNIC, storage, subnets)
    • create stamp of running instance, take stamp and create instance pool (collection of instances, manage all instances as a unit - start, stop, terminate, put in different ADs)
    • take instance pool and write autoscaling rules (initial size, max size, define thresholds)

OS Management

  • helps sysadmins automate management of Oracle Linux and Windows Server instances
  • two main characteristics of the service
    • automated patch management
    • simplified package management
  • patch management
    • managing patches which are released, usually to
      • fix bugs
      • improve performance
      • add features to the OS
    • patching manually is complex and error-prone
  • package management
    • installing, patching and removing software packages
    • hard to catalog installed and available packages and updates
  • automated patch management
    • provide single consolidated service
    • supports Oracle Linux and Windows Server
    • supports multiple OS versions
    • works on large group of instances
  • simplified package management
    • search and install/remove packages
    • inventory of installed and available packages and updates
  • common vulnerabilities and exposures lookup
    • vulnerabilities stored in CBE database
    • service looks against that database and allows installation of packages to patch vulnerabilities
  • OS management service is free
