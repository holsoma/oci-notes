Observability and Management

  • observability -> the ability to understand the internal state of a system from the outside (system needs to be properly instrumented)
  • traces, logs, events, metrics (pillars of observability)
    • metrics - numerical representation of data measured over intervals of times (eg. CPU utilization)
    • logs/events - immutable timestamped record of discrete events that happened over time, easiest to generate, but not very useful since it lacks context (this is where tracing comes in)
    • traces - understand how different services work together, tracing helps to pinpoint the cause of performance problems in a distributed system
  • extra: blackbox, whitebox monitoring
  • whole idea is to understand the state of a system, identify potential bottlenecks and patterns (using features like tracing) -> so as to have insights to take action and prevent errors

OCI Observability Services

  • monitoring service
    • several out-of-the-box and custom matrix
    • monitor health and performance of cloud infrastructure
    • configure triggers/alarms
  • logging service
    • centralized management for all logs in your tenancy
    • ingests log data from different sources, classifying them into various types (audit logs, service logs, custom logs)
  • event service
    • keeps track of changes happening to resources in tenancy
    • take automated action based on that particular change (trigger an action to convert a file into a different format once uploaded into object storage bucket)
  • only the above 3 will be covered in OCI foundations course
  • logging analytics
    • machine learning based cloud solution that helps you to aggregate index and run analytics across all log data collected across various environments
  • application performance monitoring (APM)
    • provides deeper visibility into performance of applications
  • database management
    • offers different features to monitor and manage oracle databases
  • operations insight
    • key native observability service that gives you a holistic insight into resource utilization and capacity of hosts and DBs

Monitoring Service

  • to actively collect system metrics or logs, so as to track errors as they occur
  • store metrics to observe the system and react to incident through alerts
  • eg. look at metric like CPU utilization and track it over time, then react to incidents when CPU utilization goes over a certain threshold
  • two key components: metrics and alarm
    • services like compute, database continuously emit metric to monitoring service for active monitoring
    • alarms use metrics for passive monitoring
    • alarm has a trigger action and notification method (alarm is triggered when metrics breach alarm threshold)
    • what are metrics?
      • a measurement related to health, capacity, or performance of a resource
      • logically grouped within different metric namespaces (logical containers for metrics)
      • eg. CPU utilization is under oci_computeAgent namespace (easier to operate at namespace level because we can group different matrices within these namespaces)
      • you can create custom metrics, or use service metrics (metrics emitted by services like compute, storage)
      • you can filter by dimensions (qualifiers/filters for metric data, eg. dimension of shape allows you to look at CPU utilization for this particular shape) and time period (start and end time)
      • concept of interval (time window from start to end time period) and statistic (aggregation function like interval, you can use it to look at maximum reported value of CPU utilization) when filtering
    • what are alarms?
      • alarms get triggered when they meet alarm specified triggers
      • different states - OK, firing, reset and repeat (OK means alarm is not in a firing state, firing means alarm state has been changes, has been triggered from OK to firing)
      • alarm message sent to configured topic -> message sent to all topic subscribers/subscriptions (email/SMS) -> use gets notified
      • suppression to avoid publishing messages during a specified time range (eg. during system maintenance)
  • allows you to perform active (metrics) and passive monitoring (alarms) of cloud resources

Logging Service

  • highly scalable, fully managed single pane of glass for all the logs
  • centralized and secure log management service, built on open standards
  • logs generated are searchable because they are indexed in the system, so that you can act on the log data (eg. create alarms based on log data)
  • logging integrates with service connector hub, and lets you act on logging to do things like create alarms and transport the alarms to a different service (eg. transport logs to object storage, write logs to a stream)
  • what is a log?
    • an OCI resource that stores and captures log events collected in a given context
    • eg. service logs provided to you by object storage service, by default [there are two kinds of service logs for buckets]
      • one for read access events - captures download events
      • one for write access events - captures write events, upload events
    • keep in mind each service can have different log categories for resources (log categories for one service has no relationship to log categories of another service)
  • log groups
    • when you enable a log, you must add it to a log group you create
    • log groups can be thought of as logical containers for logs, to streamline log management
  • logs and log groups are searchable and transportable, but you can use IAM policy to limit access to sensitive logs (create a sensitive data log group that contains PII)
  • different kinds of logs supported by the logging service
    • service logs - emitted by OSS, VCN, Fn, DevOps (has predefined logging categories for OCI services)
    • custom logs - emitted by customer applications running on OCI, other cloud providers and on-prem apps (custom apps can ingest logs by using API or Unified monitoring agent)
    • audit logs - related to audit events emitted by OCI audit service, accessed from same console from logging service (immediate usability, out-of-the-box)
      • calls to all OCI API endpoints are recorded as log events
      • every OCI service supports logging by audit

Event Service

  • create automation based on the state changes of resources throughout your tenancy (services emit events for resources/data)
  • for eg. OSS emits events for buckets and objects (bucket create, bucket update, object create, ...)
  • from these events, you create rules
  • event service has a concept to specify an action or a trigger when this event type occurs (creating automation based on "state changes")
  • what are events?
    • structured and schematized messages that denote a change in a resource
    • there are rules which are configuration where a user defines events that need to be created, and trigger an action if they occur
      • rules have something called an action as well, each rule must specify an action to trigger when the filter finds a matching event
      • eg. on complete of database provisioning, notify me
  • not all services supports events
