Introduction to VCN

  • region
    • a localized geographical area that is composed of one or more ADs
  • availability domain
    • one or more fault tolerant and isolated data centers that are very close to each other
    • proximity among them allows for high availability, low latency and high throughput
  • fault domain
    • a grouping of hardware and infrastructure within an AD
    • each AD has 3 FDs
  • virtual cloud network
    • a virtual representation of a physical network, similar to the one you find on-prem
    • resides in a single region
    • can have one or more non-overlapping IPv4 CIDR blocks of your choice
    • CIDR blocks/prefixes can be modified after VCNs are created (eg. /16 to /17)
    • addressing, subnets, routers, firewalls, route tables, connectivity
    • NAT gateway
      • provides the private subnet with private access to public endpoints on the internet
      • connections can be initiated only from the subnet (resources and the open internet cannot use the NAT gateway, or any other gateway to access resources in the private subnet - one way access)
    • service gateway
      • provides the private subnet with private access to supported Oracle services within the region
    • local peering gateway
      • communicates VCNs in the same region
    • dynamic routing gateway
      • connect your existing on-prem network to your VCN
      • provides a single point of entry for remote network paths coming into VCN

CIDR

  • classless inter-domain routing
    • method for allocating IP addresses and IP routing
  • virtual flat network typically /16
  • subnets in virtual flat network typically /24
  • network ID, first host and broadcast address are reserved

RFC 1918

  • networking concept, private IPv4 addresses that are recommended to apply to your VCN and subnets
  • a collection of IPv4 address ranges that should be reserved in private network
    • no resource in the open internet will ever have an IP address from this range publicly
  • recommended to stick with RFC 1918 because if you use an address that had already been implemented in the internet, and you want to reach the resource from that VCN, it will not be possible (a routing conflict is created)
    • private resources cannot be reached over the public internet directly

Introduction to Subnets

  • VCN spans all 3 availability domains
  • subnets can be either AD specific, or regional
    • load balancer or VM that belongs to an AD specific subnet is AD specific
    • load balancer in regional subnet is not affected by one AD going down
  • each subnet has a continuous range of IPv4 CIDR, and subnets cannot have overlapping CIDRs
  • a subnet can grow or shrink after creation (/24 → /25, making it smaller)
    • be careful that it is not going to overlap with something you already created
  • instances you place in a subnet gets its private IP from that subnet
  • the virtual network interface card (VNIC) determines how resources can connect with the endpoints of other resources inside or outside the VCN
    • you still need to establish perms on security list / network security group though
  • can be private or public
    • private subnets
      • resource can only have private IPs
    • public subnets
      • public IP is optional

Security Lists

  • a common set of firewall rules associated with a subnet and applied to instances launched inside the subnet
    • these rules specify the type of traffic allowed in and out of the subnet (ingress/egress)
  • you can use default security list or create your own to bypass it
  • rules can be stateful or stateless
    • stateful - only need to configure one of the directions, either ingress or egress
      • if you allow permission into port 80, you don’t need to define out of port 80, where it’s going to (response traffic automatically allowed)
      • default security list rules are stateful
    • stateless - requires creation of stateless egress rule to allow automatic response traffic (you need to configure both directions)
      • ideal for scenarios with a large number of connections
    • special cases like ICMP protocols that require an egress rule regardless of state option (otherwise ICMP will not work)
  • is enforced at the VNIC level , VNIC is the one attaching to the subnet
  • each subnet can have up to five security lists
    • default security list has three rules
      • SSH enabled for port 0, anyone can access port 22, SSH
      • two ICMP rules for troubleshooting (not echo, not pinging)
    • manually enable IPv6 traffic, or any other services

Network Security Groups

  • unlike security lists, they are not created by default
  • update NSG through attached VNICs panel
  • provides a virtual firewall for a set of cloud resources that all have the same security posture, BUT consists of rules that apply only to set of VNICs of your choice in a single VCN (unlike security lists)
  • in fact, Oracle recommends NSG over SLs
  • you can use a NSG to create exceptions to a SL
  • when you create a subnet, you NEED to add it to a security list, but the security list can be empty
    • you can use SLs or NSGs by themselves or together
  • NSGs have stateful and stateless statuses as well
    • you can create a stateless rule in the SL, then have the matching stateless rule in the NSG!

Bastion

  • bastion as a service
  • if you need to connect resources from the open internet (resides in a private subnet), you can use:
    • a bastion server that you create on a compute instance (serves like a jump box from a public subnet into resources in your private subnet) — not optimal
      • you need to create a compute instance
      • it presents danger of being accessed from an external entity
    • Oracle’s bastion service

What is the OCI Bastion Service?

  • a service that provides ephemeral access (resources are not always available) into your resources
  • customers access target resources via OCI bastion in a secure and time-bound manner
    • incoming SSH connections can be restricted to certain IPv4 address ranges or even a specific IP
  • everything is recorded through the Event and Audit service
  • OCI bastion service is integrated with IAM (you can grant permissions for those users accessing your resources in a private subnet)
  • it is FREE (bastion service, not bastion server since you have to pay for that compute instance)

What can I access with OCI Bastion Service?

  • OCI Bastion is built on to pf OpenSSH/SSH
    • anything that works over SSH can work via OCI Bastion

Bastion Works with Different Networking Architectures

  • you can put bastion service in same subnet or in a different subnet
    • in a bastion-only private subnet
    • or in same subnet where your resources reside

Internet Gateway

  • OPTIONAL virtual router that connects the VCN with the internet
    • allows connectivity into the open internet
  • egress and ingress connections from the VCN or to the VCN (you need a public IP on both ends, obviously!)
    • hence, resources that use it need to be in a public subnet and have a public IP
  • for the gateway to work, you also need a route rule in a route table, with the appropriate permissions for the connection to establish (SL or NSG)
  • the IG can only be used by resources in the VCN
    • NOT by another locally peered VCN
    • NOT by another resource connected to your VCN from an on-prem DRG
    • cannot have a transit routing
    • needs to be a resource in the same subnet!
  • one IG per VCN only

NAT Gateway

  • network address translation gateway
  • networking technique to give an entire private network or private subnet (without assigning each of the hosts a public IP)
  • host can initiate connections to the internet and receive replies, but cannot receive inbound connections initiated from the internet
  • NAT gateways are highly available and support TCP, UDP and ICMP ping traffic
  • there is a limit on the amount of NAT gateways you can have in a VCN (it is typically one)
    • max amount of concurrent connections to a single destination address is 20,000 (the 20,001th connection will need a second NAT gateway)
    • you can have up to 64 private IP addresses
  • similar to an IG, the NAT gateway can only be used by resources inside the VCN (peering into a VCN and DRG not supported)

Service Gateway

  • just like IG and NAT gateway, service gateway can be created internally, from the VCN
  • service gateway provides a secure path to the Oracle services network without leaving OCI (services like OSS and ADW)
    • these services usually have a public IP and are accessed through the open internet
  • service gateway will grant access to Oracle service network through either FastConnect, or through your site-to-site VPN (unlike IG and NAT gateway whcih are only available to resources in VCN)
    • this means that your on-prem network can leverage FastConnect to reach the Oracle services network as well
      • FastConnect has predictable latency, predictable bandwidth (unlike open internet)
  • service gateway is regional (enables access to your VCN’s region only)

Local Peering Gateway

  • process of connecting two virtual networks that are in the same region
    • so their resources can communicate with each other using their private IP addresses without routing the traffic over the internet or through on-prem network
  • a typical local peering includes:
    • two VCNs with non-overlapping CIDRs in the same region
    • a local peering gateway (LPG) on each VCN in the peering relationship
    • a connection between these two LPGs
    • supporting route rules to enable traffic to flow over the connection and only to and from select subnets in the respective VCNs (if desired)
    • supporting security rules to control the types of traffic allowed to and from the instances in the subnets
  • DRG v2 allows two VCNs to communicate with each other through the DRG remote peering connection
  • DRG connection vs local peering connection
DRGLPC

    • you have a DRG in the middle

Route Tables

  • you can use/edit the default route table or create a new one to supplement the VCN
    • new route table for private subnet, while public subnet uses default route table
  • traffic within VCN does not require route table, it is automatically handled by VCN local routing
    • no route rules are required to enable routing
    • however, you still need permission to access those resources! (for eg. a compute node still needs permission to reach a database)
  • if there are overlapping rules in a route table, the most specific rule is used
  • logically, if there is no matching rule, the packet is dropped
  • IPv6 addressing is supported for all commercial and government regions
  • each route rule has a destination CIDR block and a route target for traffic that matches that CIDR
    • exception for the service gateway (route target), you can define ‘object storage’(destination CIDR)
