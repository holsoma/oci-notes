Connectivity Options

  • FastConnect, site-to-site VPN, public internet

Routing Protocols

  • if the router for the different paths are the same, Oracle uses the shortest path
  • FastConnect - supports border gateway protocol (BGP) only
  • Site-to-site VPN - supports static routes and BGP
  • BGP - offers asymmetric routing (packets take one path to go form source to destination and a different path to return)
    • if you have two FastConnect links, traffic might originate form one link and return to on premises via the other link
    • there are ways to enforce a path that remains the same going in and returning from on premises to OCI
    • for public peering, the public autonomous system number and public registered prefixes are needed (CIDR blocks, public CIDR blocks)
      • OCI’s AS# is 31898
    • BGP prefix limit
      • public virtual circuits: 200 prefixes
        • if you want to bypass the internet to access Oracle services network, you can use FastConnect and service gateway (establish a public virtual circuit)
      • private virtual circuits: 2000 prefixes
        • cloud networks will have connectivity into on premises resources

Dynamic Routing Gateway

  • offers you the possibility of creating a global network from your on-premises site
    • if you have FastConnect/site VPN enabled in one of the regions, all of the other OCI regions are available to your on-premises resources
    • you can expand global network by attaching more FastConnects on other regions
  • network scalable
    • a single DRG can now have up to 300 virtual cloud networks attached to it
  • powerful remote connectivity
    • you can remote peer to DRGs in another cloud account
    • DRGs can be in different regions, or within one region and across tenancies
  • you can attach more than one VCN to a DRG, to allow for simplified routing
    • different route tables and policies are assigned to each network resource (the attachments you add into the DRG) for fine-grained a fine-grained routing mechanism
  • equal cost multi-path routing (ECMP)
    • off by default, but you can turn it on
  • simplified configuration
    • route across two VCNs by hopping to the same DRG that your VCN is attached to
    • bypass remote peering connections (different region) and local remote peering connections (same region)
  • high availability
    • connectivity from on premises to many regions through one FastConnect, but you can establish two FastConnects, one that acts as a backup for the other FastConnect
  • increased scale
    • DRG used to only allow you to hvae one VCN attached to it, now you can attach up to 300
  • complex routing
    • you can have fine grained routing mechanisms that allow you to connect or interconnect resources attached to your DRG

DRG Attachments

  • VCN attachments
  • RPC attachments
    • can be in the same region and across tenancies as well
  • IPSEC_TUNNEL attachments
    • can have more than one
    • redundancy is recommended when connecting to on-prem (2 IPSEC, 2 VIR_CIRCUIT, or one each)
  • VIRTUAL_CIRCUIT attachments
    • can have more than one

DRG v2

  • the above features have been introduced to DRG a year ago (2021)
  • DRGs prior to v2 are called legacy DRGs
    • if you see a message prompting you to upgrade, you have a legacy DRG
    • be mindful of the disruption to your services when upgrading (unless you have redundancy)
      • upgrade is irreversible
  • supports connectivity from on-prem to multiple regions with a FastConnect or VPN Connect terminating to a single region
  • redundancy is crucial
    • deploy multiple FastConnect circuits to one region, or in multiple regions
  • support to access Azure from multiple regions
  • ability to see routes received
    • allows you to utilize your routes and configure your VCNs

packet arrives from VCN (originates within OCI) → reaches DRG → route lookup performed on route table→ terminates other attachment, goes to on premises

  • import policies associated with route tables, route tables associated with attachments, creating a configuration structure designed to scale through large tenancies

Import Policies and Route Tables

  • you can have simplified routing with DRG through attachments or complex routing if you scale and manage all the attachments through the route tables and the imports of the policies

DRG Remote on Ramp/Transit Routing

  • you have one on-prem location that has one FastConnect (FastConnect or IPSec going from on-prem to OCI region), other OCI region is virtually available through a remote peering connection (leveraging OCI backbone)
  • this DRG has three attachments
    • a VPN attachment
    • a remote peering connection attachment
    • a FastConnect or IPSec connection attachment
  • you can add another FastConnect to provide redudancy

DRG ECMP (Active/Active) Support

equal cost multi-path routing - packets towards a single destination IP address are load-balanced over multiple best paths with equal metrics.

  • ECMP allows you to increase the capacity of your connectivity
    • two 10GB FastConnect connections can be combined as an active/active configuration with 20GB of usable capacity
  • ECMP supports the bundle of up to eight IPSec or FastConnect connections for active/active load sharing
    • either/or, you cannot mix both of them
  • ECMP is enabled on a per route table basis if something goes down in the back end

DRG and L3 Firewall

  • you can leverage a firewall
  • a route table allows you to terminate into a private IP (belonging to a compute instance or a LB)
  • on a compute instance you can install whatever you want, including layer 3 firewalls
    • you can then configure your firewall to be a termination point for your DRG (establish transitive routing from VCN to on-prem through FastConnects or IPSec)
    • you then have another route table that will inspect all traffic on the L3 firewall determine routing scheme
      • route tables are used to direct traffic via a specific VCN and the DRG-attachment forces traffic to the network virtual appliance (NVA)

DRG Route Tables

  • when packet enters DRG, it is routed using rules in route table that is assigned to the attachment (there are four types of attachments)
