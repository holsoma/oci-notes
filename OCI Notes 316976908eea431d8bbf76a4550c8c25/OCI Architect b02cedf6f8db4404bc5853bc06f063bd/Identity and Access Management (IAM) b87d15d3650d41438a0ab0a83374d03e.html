IAM Introduction

What is IAM?

  • a way for us to segregate users, give fine-grained access control
  • each user is assigned one or more predetermined roles, each role with a set of permissions
  • two main concepts, authentication AuthN (who someone is or claims to be) and authorization AuthZ (permissions, what someone is allowed to do)
  • logical containers called identity domains to separate users
  • logical containers called compartments to separate resources
  • every resource you create in OCI is an object, you can identify these resources by their OCID
    • ocid1.<RESOURCE TYPE>.<REALM>.[REGION].[.FUTURE USE].<UNIQUE ID>

Key Terms

Identity Domains

  • represents a user population in OCI and associated configurations and security settings
  • think of it as a container for managing users, roles, federation, SSO and other things like MFA
  • you can construct IDs to separate
    • production and development environment
    • employees and non-employees
  • you can switch identity domain type and create additional identity domains
  • IDs are resources which reside inside of a compartment

Free Identity Domain

  • default identity domain to manage access to all OCI resources
  • good if they don’t want to create anything additional

Oracle Apps Premium

  • helps manage SaaS, PaaS, GBU applications
  • helps manage on prem applications (JD Edwards, PeopleSoft, Oracle Linux, Oracle DB, etc.)
  • supports hybrid IAM use cases (on prem + cloud)

Premium Domain

  • highest SKU
  • you can use Oracle or non-Oracle apps
  • supports hybrid IAM use cases (on prem + cloud)

External User

  • used for consumer and non employee use cases
    • give contractors access
    • manage application that can scale to millions of users
  • features like social login, self-service, password management, etc.
  • some support for hybrid features

AuthN

What is a Principal?

  • an IAM entity that is allowed to interact with OCI resources
  • can be users, resources or services themselves
    • instance principal - a compute instance that lets instances and applications running on those instances make API calls against other OCI services

Groups

  • collection of users who all need the same type of access to resources
    • network-user group - for users who create networks
    • network-admin group - for users who administer or manage networks

AuthN Mechanisms

  • username + password
  • API signing keys (used when you call the OCI CLI, SDK or when you write your applications)
    • RSA key pair, public and private key
  • auth tokens
    • oracle-generated token strings
    • each user can generate up to two auth tokens that don’t expire and can authenticate third party APIs
    • use case: Docker registry, because it doesn’t understand OCI signature-based authentication

AuthZ

  • done through IAM policies
  • starts with ALLOW <subjects> to <action> in <placement> WHERE <condition>
  • by default, everything is denied
  • no deny policy by default, because everything is denied
  • remember: you write policies on groups, NOT users
  • you can write policies for dynamic groups (links specific instances in the compartment and provides security and enables you to manage keys on client side rather than server) as well to grant a particular service

Subjects

  • subject’s clause is your principal actor, you can define it
    • by their membership, role requirements (group of administrators) using name or group id
    • for a particular service (OSS)
      • you would do this to allow replication (copy of OSS object from one region to another region)
  • you can use subject’s clause with “any-user”, meaning that anybody who is part of that account can do something
  • more than one group can be named in subjects element
  • you need to prefix your subjects with an identity domain, if you don’t do it, it will go to the default ID

Actions

  • inspect, read, use and manage (access level increases as you go down the hierarchy)
    • inspect - observe, enumerate and monitor, without access to confidential information
    • read - access without altering resources
    • use - modify pre-existing resources (re-encrypt objects using a different key version)
    • manage - do anything to the resource kind (create and delete objects)
  • resource type (second part of action clause)
    • some resource types don’t have aggregate resource types (like identity)

Common IAM Policies

  • common policies can be found on the OCI documentation website
    • let network admins manage cloud network
    • let users launch compute instances

Compartments

  • your account/tenancy is sometimes referred to as root compartment
  • compartments are logical containers (not physical) that does not need to be service specific
    • you can have compartments for projects, applications, whichever way you want to organize
    • isolate and control access
  • each resource belongs to a single compartment
  • you can write policies for users, grouped in groups, to control access to the compartments (authZ)
  • resources in different compartments can interact with each other
    • VCN can be in one compartment, VM can be in another
  • resources from multiple regions can be in the same compartment (a compartment is global)
  • compartments can be nested, 6 levels deep

Compartment Quota and Budget

  • you can set quotas or budgets on a compartment
    • if budget is exceeded, you will get notified
  • compartment quotas are similar to service limits, biggest difference is that service limits are set by Oracle, while compartment quotas are set by administrators using policies to flexibly allocate resources
  • quota policies
    • set max limit, or set to zero so resources cannot be created
    • restrict all users with access to this compartment to provision compute instances (users with access to other compartments will still be able to provision)
  • budgets let you know how much resource consumption is happening and helps you make sure usage is not exceeding a particular budget you have
    • budget amount can be an absolute number or a percentage of the budget
    • send email or alert when budget is exceeded

Policy Inheritance and Attachment for Compartments

  • compartments inherit any policies form their parent compartment
    • built in policy for Administrators applies to all child compartments, so Admins can do anything in the compartments in the tenancy
    • capability to manage any of the compartments downstream
  • attachment of policy to a compartment
    • compartment A > B > C
    • if you want to write a policy to manage networks in C, you can write it in C or B, but you have to provide a complete path when you write it in A (because A can only see immediate children, can’t see two level children)
    • attachment of policies to a compartment also means that users will not be able to delete or change policies upstream, if they don’t have access

Conditional Policies

  • placement is your compartments or tenancy, where you can write advanced policies by putting these conditions where you can use the keyword WHERE and the condition is evaluated to true/false or not applicable
  • conditions allow more complicated and advanced fine-grain access control
    • you use variables when writing conditional statements
    • variables are hierarchically named, prefixed accordingly with either request or target followed by a period

    • you can use strings and patterns in conditions

Network Sources

  • mechanism to control access based on originating IP addresses
  • a two step process
    1. define an object, a network source (a set of IP addresses, either specific VCN addresses, public IPs or both)
    1. write a policy (specify a group of users that can access the object, using the request.networkSource.name variable in a condition)

Tag-based Access Control

  • you can use tag-based access control instead of duplicating policies to manage groups or resources spanning multiple compartments
  • scope policies based on the tags, which are applied to a resource (can be requesting resource or target of the request)
  • TBAC = conditions + a set of variables
  • tags have a key and value (namespace.value)

Dynamic Groups

Types of Principals

  • three types of principals
    • infrastructure principals
      • a birth certificate
      • an IAM service feature that enables instances to be authorized actors (or principals) to perform actions on service resources
      • (eg. instance principal)
    • stacked principal
      • requesting a passport, having a birth cert
      • a service (not the infrastructure) controls a resource, specifies the intention of the resource
      • requires the infrastructure to be hosting one resource, and multiple infrastructures might host the same resource for redundancy
      • (eg. Oracle database, since it makes calls to OSS to manage backups)
    • ephemeral principals
      • a temporarily issued badge
      • temporary credentials given for a specific purpose and then it expires
      • injected identifiers which defines who the holder of a particular credential is for a short period
      • (eg. Oracle Functions, because it can make call to OSS for a short period of time)

What are Dynamic Groups?

  • dynamic because it refers to groups that have a set of matching rules to define their members, so members of the group can change dynamically (rather than adding/removing members explicitly)
  • allows infrastructure, stacked and ephemeral resource principals to be grouped as principal actors
  • policies can be written against dynamic groups for DGs to make API calls against OCI services (instance calling OSS, Functions calling OSS)

Creating Dynamic Groups

  • you can use OCID to refer to a particular instance or all instances within a compartment
  • you can specify the resource type and/or compartment
  • then you would create a policy to refer to the dynamic group
